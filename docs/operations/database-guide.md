# Database Guide

운영 DB에서 어떤 테이블이 어떤 역할을 하는지, 테스트 데이터와 삭제를 어떻게 다룰지 정리한 문서입니다.

## 주요 테이블

| 테이블 | 역할 |
| --- | --- |
| `orders` | 고객 주문/견적/예약의 중심 테이블 |
| `customers` | 고객 이름, 연락처 등 고객 정보 |
| `homes` | 주소, 상세주소, 지역 정보 |
| `quotes` | 견적 금액, 제품가, 시공비 스냅샷 |
| `payments` | 결제/입금 상태 정보 |
| `jobs` | 기사 배정, 방문, 시공 진행 상태 |
| `diagnoses` | 사진확인 접수 및 판정 데이터 |
| `media` | 주문/사진확인/작업 관련 파일 메타데이터 |
| `technicians` | 기사 정보와 활성 상태 |
| `slot_configs` | 날짜별 예약 슬롯 설정 |
| `events` | 방문/전환/조회 이벤트 로그 |
| `app_configs` | 운영 설정값 |

## 주문 관리 기준

관리자 주문 목록은 기본적으로 `orders`를 중심으로 고객, 주소, 견적, 결제, 작업 정보를 조합해 보여준다.

운영 중 주문을 삭제해야 할 때는 바로 완전삭제하지 않는다.

1. 관리자에서 휴지통으로 이동
2. 휴지통에서 한 번 더 확인
3. 복구 필요 여부 확인
4. 완전삭제

테스트 주문은 운영 주문과 섞이지 않도록 `is_test` 기준으로 분리한다.

## 사진확인 관리 기준

사진확인 접수는 `diagnoses`에 저장된다.

운영자가 봐야 할 핵심 정보:

- 접수번호
- 고객 이름/연락처
- 주소와 작업 가능 지역 확인 여부
- 요청 서비스
- 업로드 사진
- 판정 결과 또는 상담 필요 여부

사진확인 데이터를 주문으로 전환할 때는 고객에게 안내 후 진행한다.

## 테스트 데이터 처리

운영 중 테스트가 필요한 경우:

- 가능한 관리자 테스트 흐름을 사용한다.
- 실제 고객 데이터와 구분되도록 테스트 표시를 유지한다.
- 테스트가 끝난 뒤 휴지통 이동 또는 테스트 목록에서 정리한다.

Supabase에서 직접 삭제해야 하는 경우:

- 삭제 대상 테이블과 row 수를 먼저 확인한다.
- 실제 운영 예약/주문이 없는지 확인한다.
- `orders`, `customers`, `homes`, `quotes`, `payments`, `jobs`, `media`, `diagnoses` 연결 관계를 고려한다.
- 삭제 전후 관리자 화면과 Supabase row 수를 다시 확인한다.

## 운영 DB 점검 항목

| 점검 | 기준 |
| --- | --- |
| 새 주문 생성 | `orders`, `customers`, `homes`, `quotes`, `payments`가 정상 생성 |
| 사진확인 접수 | `diagnoses`와 관련 `media`가 정상 생성 |
| 주문 조회 | 이름/전화번호로 최신 주문 조회 가능 |
| 입금 확인 | 관리자에서 결제 확인 후 상태 변경 가능 |
| 예약 관리 | 기사 수와 슬롯 설정 기준으로 예약 가능 시간 표시 |
| 휴지통 | 복구와 완전삭제가 분리되어 동작 |

## 금지 사항

- 운영 중 전체 테이블 삭제 금지
- access token, 고객 연락처, 주소 원문을 문서나 채팅에 남기지 않기
- service role key를 브라우저, 문서, 공개 저장소에 노출하지 않기
