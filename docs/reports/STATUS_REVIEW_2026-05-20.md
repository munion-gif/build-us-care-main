# Buildus Care 현황 정리

작성일: 2026-05-20  
기준 배포: `https://builduscare.co.kr`, `https://buildus-care-flow.vercel.app`  
기준 자료: 현재 코드, 최근 배포 결과, `buildus_care_data.xlsx`, `빌드어스_데이터수집매뉴얼_v1_2026-05-07.xlsx`, 기존 QA/진행 문서

## 1. 결론

현재 Buildus Care는 고객이 사진확인 또는 견적 흐름으로 들어와 주문/예약/결제/상태조회까지 이어지는 MVP 운영 구조는 거의 갖춘 상태다.

다만 실제 운영 관점에서는 아직 `실알림`, `실결제 운영 전환`, `데이터 품질 관리`, `관리자 운영 SLA`, `권한/보안 고도화`가 남아 있다. UI/UX는 브랜드 방향과 주요 모바일 화면은 정리됐지만, 견적 이후의 전체 입력 흐름과 관리자 화면은 운영자가 매일 쓰는 기준으로 한 번 더 밀도 조정이 필요하다.

## 4. 엑셀 기준 Stage 현황

`빌드어스_데이터수집매뉴얼_v1_2026-05-07.xlsx` 기준으로 보면 현재는 Stage 0~1은 운영 가능한 수준이고, Stage 2는 구조 중심, Stage 3 이후는 후순위다.

| Stage | 엑셀 기준 | 현재 상태 |
| --- | --- | --- |
| Stage 0 | customers, homes, orders, quotes, payments, jobs, media, feedbacks, technicians, materials P0 스켈레톤 | 거의 완료 |
| Stage 1 | inspections, warranty_cases, 표준 사진, 기본 검수 | 대부분 완료 |
| Stage 2 | events, UTM, 광고 채널, 자재 페어링, ROI 측정 | 부분 완료 |
| Stage 3 | memberships, 코호트, NPS 자동 집계, 시공자 매칭 | 미완료에 가까움 |
| Stage 4~5 | content_pages, SEO 전환, AI 학습 라벨링, 고급 자동화 | 일부 구조만 존재 |

`buildus_care_data.xlsx`는 25개 시트가 있으며 README, collection_map, data_dictionary, customers_private, homes, orders, quotes, payments, reservations, jobs, media, inspections, feedbacks, warranty_cases, notification_queue, events, sessions, materials, technicians, dim_* 및 dashboard 구조를 포함한다. 대부분 샘플/스켈레톤 데이터라 운영 데이터 누적과 품질 관리가 다음 과제다.

## 6. 지금 기준 꼭 필요한 다음 작업

1. 실알림 연결  
   현재 알림 UI는 보류/준비 상태로 정리됐지만 실제 발송은 되지 않는다. SOLAPI 또는 카카오 알림톡/SMS 채널, 발신번호, 관리자 수신 채널을 연결해야 한다.

2. 결제 운영 전환 체크  
   Toss mock 기준은 통과했지만 실결제 키, webhook 검증, 환불/취소 연결, 운영 장애 대응은 별도 체크리스트가 필요하다.

3. 관리자 “오늘 처리할 일” 큐 정리  
   사진접수, 미배정 paid 주문, 예약확정 대기, 방문 예정, 취소/A/S 요청을 한 화면에서 우선순위로 보여줘야 운영 효율이 올라간다.

4. 사진/시공 데이터 품질 기준  
   사진 앵글, 전/중/후, 문제 부위, 주변 환경, 자재 사진 같은 필드를 강제하거나 운영자가 라벨링할 수 있어야 엑셀의 AI 학습 자산 목표와 맞는다.

5. 권한/보안 고도화  
   현재 admin은 단일 세션/비밀번호 기반이다. 실운영 전에는 관리자 계정, 권한, 감사 로그, 개인정보 접근 로그, 보관/삭제 정책이 필요하다.

6. 모바일 전체 플로우 QA  
   사진확인/서비스/사례/견적은 1차 보정했지만, 견적 2~3단계, 주문상태, 취소/A/S, 기사 앱까지 실제 휴대폰 기준으로 최종 점검해야 한다.
